# delivery-time

Telegram-бот: курьер присылает ежедневный PDF-отчёт («Twoje podsumowanie»),
бот распознаёт дату и время доставки, хранит данные по дням и считает
**среднее время доставки за неделю (Пн–Вс)** — метрику, влияющую на статус аккаунта.

## Структура

| Файл | Назначение |
|------|-----------|
| `parser.py` | Разбор текста отчёта в структуру (дата, время доставки/выезда, заказы, заработок). Не зависит от Telegram. |
| `db.py` | SQLite: хранение по дням, дедуп по `(user, дата)`, недельное среднее. |
| `bot.py` | Telegram-бот (telebot): команды и обработка PDF. |
| `tests/` | Тесты парсера на обезличенной фикстуре реального отчёта. |

## Запуск локально

```bash
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env        # вписать TOKEN от @BotFather
python bot.py
```

## Тесты

```bash
python tests/test_parser.py
```

## Команды бота

- `/start` — приветствие и инструкция
- (прислать PDF) — распознать и сохранить отчёт за день + показать недельную сводку
- `/stats` — среднее за текущую неделю и статус
- `/reset` — удалить все свои данные

## Особенности логики

- **`∅` (нет времени доставки).** На сменах после 23:00 платформа не считает время
  доставки и ставит `∅`. Такие дни **исключаются** из среднего (не обнуляются и не
  подменяются временем выезда). Время выезда считается всегда.
- **Дедуп.** Повторная отправка отчёта за тот же день обновляет запись, а не задваивает.
- **Среднее за неделю** = средневзвешенное дневных средних по числу заказов.

## Деплой (always-on)

Бот работает на long-polling (`infinity_polling`), поэтому ему нужен **постоянно
запущенный процесс 24/7**. Бесплатные веб-сервисы (напр. Render free) засыпают
после простоя — для polling-бота не подходят. Нужен always-on worker.

**Переменные окружения на хостинге:**
- `TOKEN` — токен бота (задавать в настройках хостинга, НЕ в репозитории)
- `DB_PATH` — путь к файлу БД, напр. `/data/delivery.db`

**Хранение БД:** SQLite — это файл. На большинстве PaaS файловая система эфемерна
и **обнуляется при каждом редеплое**. Чтобы данные пользователей не пропадали,
примонтируй постоянный том (volume/disk) в `/data` и укажи `DB_PATH=/data/delivery.db`.

**Вариант A — Docker (подходит почти всем: Railway, Fly.io, VPS):**
```bash
docker build -t delivery-time .
docker run -d --restart=always \
  -e TOKEN=... -e DB_PATH=/data/delivery.db \
  -v delivery_data:/data delivery-time
```

**Вариант B — Procfile (Railway/Render как worker):** деплой из GitHub, тип сервиса
**Background Worker** (не Web), переменные `TOKEN`/`DB_PATH`, том на `/data`.

⚠️ Токен ранее засветился в переписке — перед деплоем сделай **Revoke** в @BotFather
и пропиши НОВЫЙ токен в переменных окружения хостинга.
